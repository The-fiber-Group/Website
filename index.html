About Us

The fiber Group is the body of authors responsible for The fiber Language and Environment Standard, or "fiber". We also develop the reference implementation of fiber as well.

I say "we", but there's only one author right now, me, Robin. I hope this changes soon.

About the fiber Standard

Fiber is an open standard for a programming language and execution environment. It describes...

  • a programming language, fiber,
  • an abstract computer architecture/model, Bran,
  • a true architecture-independent bytecode, BRISC,
  • a linking loader program, Broccoli,
  • and a linkable binary object format, BOF.

fiber has a few unique advantages compared to other programming languages which are described on the following cards.

Project Status

Right now, the fiber Language and Environment Standard is incomplete. Don't try to use it with anything serious. The Environment portion of the standard is complete. The Language portion has yet to begin.

Portability

Bran, the fiber Environment, is a self-contained, abstract, computer architecture model, similar to the ones used in C and Rust. Its instruction set, BRISC, has a near one-to-one instruction correlation with many other instruction set architectures. It has a minimal amount of abstractions, enough to guarantee proper memory alignment and data type handling across many different architectures. The BOF object format is capable not only of storing Bran programs, but any other binary format as well.

Bran provides the same "write once, run anywhere" utility of more complex abstract virtual machines. Unlike other abstract environments, the entire Bran architecture is facilitated by the glorified linking loader, Broccoli. The advantage of Bran is twofold. It is much cheaper and faster to implement: Only Broccoli and a BOF object library for system interaction are required. Additionally, once Broccoli is implemented, any and all existing Bran programs in the world can be run on the host system with native performance.

Reliability

A unique property of the fiber programming language is that it has a built in proof syntax. With more and more critical infrastructures relying on software, it has become necessary to formally verify the behavior of critical software components in a system. Unfortunately, there is a disconnect between the systems used to verify a component and the programming languages used to implement it. There is no reliable way to verify that the implementation or even the programming language used for implementation will conform to the verification it was based off of and this has become a sticky situation for many projects.

fiber solves this by providing both automated proving capabilities as well as an imperative programming language as part of the same language environment. Logic syntax similar to what is found in automated theorem proving languages can be included as annotation to any piece of fiber syntax. Directive statements can than be used to engage fiber's proof system to verify any aspect of a program. This closes the gap between proofs and programs because the types, functions, and variables that would normally be locked inside a foreign proof environment are now part of the implementation language and part of the real, executable program being verified.

Stability

fiber is a standard first, and an implementation second. This means that any change or addition must first go through a long and rigorous process of being vetted and written into the standard. There is no volatility that comes from systems developed by experimenting with the implementation first. The standard documentation is also controlled by a limited number of people. fiber is only "open" regarding its use, not its contributions. This spares fiber from degradation by attempts to chase after temporary and stylistic "fads".

The fiber programming language also provides a rich type system alongside its proof syntax. Program state can be controlled with a fine granularity using dependent and substructural datatyping methods. This helps to avoid resource handling errors that would otherwise cause memory leaks, security vulnerabilities, and other undefined behavior.

Contributing

Outside contributions are welcome. If you would like to contribute to the standard documentation, please read (Authoring, 4). You can submit proposals to the email listed in Links. Please keep in mind that very few external proposals will be accepted.

If your proposal is accepted into the standard documentation, your name as it appears on the proposal will be included in the list of external authors at the bottom of the standard document.

Becoming an Author

If you would like to join the fiber Group as an author, send a request with a resume of your roles in previous projects of a similar nature. Please keep in mind, joining the fiber Group as an author is an informal per-case decision made by existing authors. The fiber Group is very much intended to be small and exclusive simply by nature of the work being done.
