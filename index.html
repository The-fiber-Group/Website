About The Fiber Group

The Fiber Group is the body of authors responsible for The Fiber Language and Environment Standard, known as "Fiber". We also develop the reference implementation of Fiber as well.

About the Fiber Standard

Fiber is an international open standard for a programming language and execution environment. It describes...

  • a programming language, Fiber,
  • an abstract computer architecture/model, Bran,
  • a true architecture-independent bytecode, BRISC,
  • a program loader, Broccoli,
  • and a binary object format, the Bran Object File (BOF).

Fiber has a few unique advantages compared to other programming languages which are described on the cards below.

Project Status

Right now, the Fiber Language and Environment Standard is incomplete. Don't try to use it with anything serious. The Environment portion of the standard is almost finalized. The Language portion is still under heavy development.

Portability

Innovation in processing hardware is starting to trend towards special purpose computing, where the focus was once general purpose applications. If this continues, the computing industry will be flooded with many unique architectures. General purpose computer languages are not designed to handle this change. Most simply require too much work to be ported to a high number of architectures. This is partly why embedded system manufacturers are known to include Forth with their systems; it's just cheaper. Fiber is designed to be equally as portable as an assembler program, while still providing high abstraction programming utilities.

The portability of Fiber lies within its execution model, Bran. Bran is facilitated by two entities, the BOF format and the Broccoli program loader. BOF files store programs using an architecture-independent instruction set called BRISC. BRISC instructions have one-to-one counterparts in hardware architectures, allowing for easy translation. During the loading process, Broccoli will translate these instructions to the native instruction set of the hardware and then execute the program as if it were written in native assembly syntax.

This simplicity allows Bran to provide the same "write once, run anywhere" portability without the complexity of more abstracted virtual machines. The advantage of Bran is then twofold. It is much cheaper and faster to implement: Only Broccoli and an optional BOF object library for operating system interaction are required. Additionally, once Broccoli is implemented, any and all existing Bran programs in the world can be run on the host system with native performance.

Reliability

A unique property of the Fiber programming language is that it has a built in proof syntax. With more and more critical infrastructures relying on software, it has become necessary to formally verify the behavior of critical software components in a system. Unfortunately, there is a disconnect between the systems used to verify a component and the programming languages used to implement it. There is no reliable way to verify that the implementation or even the programming language used for implementation will conform to the verification it was based off of and this has become a sticky situation for many projects.

Fiber solves this by providing both automated proving capabilities as part of an imperative programming language as part of the same language environment. Logic syntax similar to what is found in automated theorem proving languages can be included as annotation to any piece of Fiber syntax. Directive statements can then be used to engage Fiber's proof system to verify any aspect of a program. This closes the gap between proofs and programs because the types, functions, and variables that would normally be locked inside a foreign proof environment are now part of the implementation language and part of the real, executable program being verified.

Stability

Fiber is a standard first, and an implementation second. This means that any change or addition must first go through a long and rigorous process of being vetted and written into the standard. There is no volatility that comes from systems developed by experimenting with the implementation first. The standard documentation is also controlled by a limited number of people. Fiber is only "open" regarding its use, not its contributions. This spares Fiber from degradation by attempts to chase after temporary and stylistic "fads".

The Fiber programming language also provides a rich type system alongside its proof syntax. Program state can be controlled with a fine granularity using dependent and substructural datatyping methods. This helps to avoid resource handling errors that would otherwise cause memory leaks, security vulnerabilities, and other undefined behavior.

Contributing

Feature Proposals

Outside contributions are welcome. If you would like to contribute to the standard documentation, please read (Authoring, 5). You can submit proposals to the email listed in Links. Please keep in mind that not every external proposal will be accepted, but including as many details as you can will definitely help.

If your proposal is accepted into the standard documentation, your name as it appears on the proposal will be included in the list of external authors at the bottom of the standard document.

Joining The Working Group

If you would like to become a volunteer member of the Fiber Group, email a request with a description of your roles in previous projects of a similar nature. Please keep in mind, joining the Fiber Group as an author is an informal per-case decision made by the existing authors. The Fiber Group is primarily a decision-making body and is very much intended to be small and exclusive.

Donating

Fiber is a volunteer effort at this time and we do not yet have an established system for receiving financial contributions that guarantee your money's being spent towards Fiber. If you are determined to give us money, please reach out to us and we'll see what arrangements can be made.
