The fiber Group is the body of authors responsible for The fiber Language and Environment Standard, or "fiber". We also develop the reference implementation of fiber as well.

I say "we", but there's only one author right now, me, Robin. I hope this changes soon.

About the fiber Standard

Fiber is an open standard for a programming language and execution environment. It describes...

  • a programming language, fiber,
  • an abstract computer architecture/model, Bran,
  • a true architecture-independent bytecode, BRISC,
  • a linking loader program, Broccoli,
  • and a linkable binary object format, BOF.
fiber has a few unique advantages compared to other programming languages.

Portability

The fiber Environment is separate from the programming language. Bran is a self-contained computer architecture, similar to the ones used by C or Rust, with it's own RISC-style instruction format and binary object format. The instruction set for Bran posesses a few abstractions to ensure that it's architecture-independent for most hardware. Otherwise, it has a near one-to-one relationship with many other instruction set architectures. The BOF object format is capable of containing not only Bran programs, but any other binary format including native hardware instructions.

This simplicity brings several advantages. The first is the same "write once, run anywhere" level of portability that other virtual machine-based languages aim for. The second is that it allows implementors to support the fiber programming language with less overhead. Normally, an entire compiler or abstract virtual machine would have to be written or ported for a new architecture to support a programming language. This is a big undertaking.

The fiber Environment focuses on reducing the minimum work needed to bring a language to a new architecture. The Broccoli program is a glorified linker/loader. It's only responsibility is to load BOF files, translate any Bran instructions to native instructions, and execute them. The functions Broccoli performs take much less time to implement compared to a traditional language runtime. The payoff is twofold, fiber is cheaper and faster to support, and once Broccoli is implemented, any and all existing Bran programs in the world can be run on that system with native performance.

Reliability

A unique property of the fiber programming language is that it has a built in proof syntax. With more and more critical infrastructures relying on software, it has become necessary to formally verify the behavior of critical software components in a system. Unfortunately, there is a disconnect between the systems used to verify a component and the programming languages used to implement it. There is no reliable way to verify that the implementation or even the programming language used for implementation will conform to the verification it was based off of and this has become a sticky situation for many projects.

fiber solves this by providing both automated proving capabilities as well as an imperative programming language as part of the same compilation environment. Logic syntax similar to what is found in automated theorem proving languages can be included as annotation to any piece of fiber syntax. Directive statements can than be used to engage fiber's proof system to verify any aspect of a program. This closes the gap because the types, functions, and variables that would normally be locked inside a foreign proof environment are now part of the implementation language and part of the real, executable program being verified.

Stability

fiber is a standard first, and an implementation second. This means that any change or addition must first go through a long and rigorous process of being vetted and written into the standard. There is no volatility that comes from systems developed by experimenting with the implementation first. The standard documentation is also controlled by a limited number of people. fiber is only "open" regarding its use, not its contributions. This spares fiber from degradation by the introduction of fads which has happened to other languages.

The fiber programming language also provides a rich type system alongside its proof syntax. Program state can be controlled with a fine granularity using dependent and substructural datatyping. This helps avoid resource handling errors that would otherwise cause memory leaks and security vulnerabilities.

Contributing

Outside contributions are welcome. If you would like to contribute to the standard documentation, please read (Authoring, 4). You can submit proposals to the email listed in Links. Please keep in mind that very few external proposals will be accepted.

If your proposal is accepted into the standard documentation, your name as it appears on the proposal will be included in the list of external authors at the bottom of the standard document.

Becoming an Author

If you would like to join the fiber Group as an author, send a request with a resume of your roles in previous projects of a similar nature. Please keep in mind, joining the fiber Group as an author is an informal per-case decision made by existing authors. The fiber Group is very much intended to be small and exclusive simply by nature of the work being done.
